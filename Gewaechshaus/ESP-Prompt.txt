  FEATURE-PROMPT: RFID-Tuerzugangskontrolle fuer das Gewaechshaus-Projekt

KONTEXT & PROJEKTBESCHREIBUNG
-----------------------------
Das Gewaechshaus-Projekt ist eine Smart-Greenhouse-Webanwendung, die auf einem
Raspberry Pi laeuft. Der Tech-Stack besteht aus:

- Backend: Node.js (raw HTTP-Server, KEIN Express), Datei: server-simple.js
- Frontend: Vanilla JavaScript (script.js, ~4200 Zeilen), HTML (index.html), CSS
- Datenbank: JSON-Datei (data.json) - dateibasierte Persistenz
- Authentifizierung: Session-basiert mit Cookies (scrypt-Hashing)
- Bestehende ESP32-Anbindung: WebSocket via Node-RED (ws://localhost:1880/ws)
- Bestehendes Logging: System-Logs (data.logs) + Audit-Logs (data.auditLogs)
- Port: 3001
- Sprache im Code: Deutsch/Englisch gemischt (UI ueberwiegend Deutsch)

Bestehende Struktur der data.json:
{
  zones: [],          // Hochbeete
  slots: [],          // Pflanzplaetze
  plants: [],         // Pflanzen mit Metadaten
  users: [],          // Benutzerkonten (id, username, password, role)
  logs: [],           // System-Logs (max 500)
  auditLogs: [],      // Audit-Trail (max 500)
  forumPosts: [],     // Community-Forum
  settings: {},       // Allgemeine Einstellungen
  colorScheme: {},    // Theme-Farben
  emailConfig: {},    // SMTP-Konfiguration
  emailLogs: []       // E-Mail-Versandprotokoll
}

Es existiert bereits ein Side-Quest-Projekt unter side-quests/RFID-Lock/ mit
einem Arduino-basierten RFID-Reader (read-rfid.ino) fuer den MFRC522-Sensor.
Dieses Projekt soll nun auf ESP32 portiert und vollstaendig in die
Gewaechshaus-Weboberflaeche integriert werden.


================================================================================
FEATURE-ANFORDERUNG: RFID-Tuerzugangskontrolle
================================================================================

Es soll ein vollstaendiges RFID-Tuerzugangssystem implementiert werden, das
einen ESP32 mit RFID-Reader (MFRC522) steuert, um eine Tuer zum Gewaechshaus
zu oeffnen. Die Verwaltung erfolgt komplett ueber die bestehende Weboberflaeche.


--------------------------------------------------------------------------------
1. ESP32 FIRMWARE (Arduino/PlatformIO - C++)
--------------------------------------------------------------------------------

1.1 Hardware-Setup:
    - ESP32 DevKit V1 (oder kompatibel)
    - MFRC522 RFID-Reader ueber SPI
    - Relais-Modul oder elektrischer Tueroffner (GPIO-Steuerung)
    - Optionale Status-LED (Gruen = Zugang gewaehrt, Rot = Abgelehnt)
    - Optionaler Buzzer fuer akustisches Feedback

1.2 Netzwerkkommunikation:
    - WiFi-Verbindung zum lokalen Netzwerk
    - AUSSCHLIESSLICH HTTP-Client (HTTPClient.h) fuer REST-API-Kommunikation
      mit dem Gewaechshaus-Server. KEIN WebSocket verwenden (siehe 1.4).
    - Regelmaessiger Heartbeat/Ping an den Server (alle 30 Sekunden)
      POST /api/rfid/heartbeat
    - Der ESP soll seinen Status (online/offline, letzte Aktivitaet,
      Firmware-Version, IP-Adresse, WLAN-Signalstaerke) an den Server melden

1.3 RFID-Kartenverarbeitung:
    - Karte wird gescannt -> UID wird ausgelesen
    - ZUERST lokale Pruefung gegen gespeicherte UIDs im NVS (Non-Volatile
      Storage) des ESP32 -> Damit die Tuer auch bei Server-Ausfall funktioniert
    - ZUSAETZLICH Validierung ueber HTTP-Request an den Server:
      POST /api/rfid/access {uid: "AA:BB:CC:DD", device_id: "esp32-door-01"}
    - Bei Zugang gewaehrt: Relais fuer konfigurierbare Zeit aktivieren (Standard:
      3 Sekunden), gruene LED, kurzer Buzzer-Ton
    - Bei Zugang verweigert: Rote LED blinkt, langer Buzzer-Ton
    - Jeder Scan-Vorgang (erfolgreich oder nicht) wird an den Server gemeldet

    OFFLINE-FIRST KONFLIKTAUFLOESUNG:
    - Im Normalfall wird die Tuer per lokaler Kartenliste geoeffnet und der
      Zugang parallel per HTTP an den Server gemeldet. Die Server-Antwort wird
      NICHT abgewartet, um Latenz zu vermeiden.
    - Ist der Server nicht erreichbar, funktioniert die Tuer trotzdem anhand
      der lokalen Kartenliste. Die Scan-Events werden in einem lokalen
      Ringpuffer (max 50 Eintraege) zwischengespeichert und beim naechsten
      erfolgreichen Server-Kontakt nachgemeldet (POST /api/rfid/access-bulk).
    - Bei JEDER erfolgreichen Serververbindung (Heartbeat, Access, Polling)
      gleicht der ESP seine lokale Kartenliste mit dem Server ab:
      GET /api/rfid/device/{id}/cards liefert die aktuelle Kartenliste.
      Der ESP ueberschreibt seine lokale Liste vollstaendig mit der
      Server-Antwort. So werden serverseitig deaktivierte oder geloeschte
      Karten sofort auch lokal entfernt.
    - Wurde eine Karte serverseitig deaktiviert waehrend der ESP offline war,
      kann diese Karte bis zum naechsten Server-Abgleich noch lokal Zugang
      erhalten. Dies ist eine bewusste Design-Entscheidung zugunsten der
      Verfuegbarkeit (Tuer funktioniert immer) gegenueber der absoluten
      Konsistenz. Im Access-Log wird dies ueber method: "local" kenntlich
      gemacht.

1.4 Anlernmodus:
    - Der Server kann den ESP32 in einen "Anlernmodus" versetzen.
    - KOMMUNIKATIONSWEG: Ausschliesslich HTTP-Polling. Der ESP fragt alle
      2 Sekunden GET /api/rfid/device/{id}/mode ab, um Modus-Aenderungen
      zu erkennen. KEIN WebSocket verwenden - die bestehende WebSocket-
      Verbindung ueber Node-RED dient ausschliesslich den Sensor-ESPs.
      Eine Mischung beider Ansaetze wuerde die Komplexitaet unnoetig erhoehen.
      HTTP-Polling ist einfacher, robuster und fuer diesen Anwendungsfall
      (Latenz von 2 Sekunden ist akzeptabel) voellig ausreichend.
    - Im Anlernmodus: Naechste gescannte Karte wird als neue Karte registriert
    - Die UID wird sowohl lokal gespeichert als auch an den Server gesendet:
      POST /api/rfid/device/{id}/register-card {uid: "AA:BB:CC:DD"}
    - Anlernmodus laeuft nach konfigurierbarem Timeout automatisch ab (Standard:
      60 Sekunden)
    - Visuelles Feedback: Gelbe/blaue LED blinkt waehrend des Anlernmodus
    - Im Normalmodus (kein Anlernmodus aktiv) reduziert der ESP die Polling-
      Frequenz auf alle 5 Sekunden, um Ressourcen zu schonen.

1.5 Kartensynchronisation:
    - Beim Start des ESP32: Alle Karten vom Server laden und lokal speichern
      GET /api/rfid/device/{id}/cards
    - ESP pollt die Kartenliste in regelmaessigen Abstaenden (alle 5 Minuten)
      per HTTP-Request. Aenderungen auf dem Server (Karte hinzugefuegt,
      entfernt oder deaktiviert) werden so innerhalb von 5 Minuten uebernommen.
    - Zusaetzlich wird bei jedem Heartbeat (alle 30 Sekunden) ein kompakter
      Hash/Versionszaehler der Kartenliste mitgesendet. Aendert sich der Hash,
      laedt der ESP die vollstaendige Kartenliste sofort nach.

    NVS-SPEICHERLIMIT:
    - Der NVS (Non-Volatile Storage) des ESP32 bietet nur ca. 15-20 KB
      nutzbaren Speicher. Pro RFID-Karte werden ca. 7 Bytes UID + 1 Byte
      Status gespeichert, plus NVS-Overhead (~40 Bytes pro Eintrag).
    - Maximales Limit: 50 Karten pro ESP32-Geraet. Dieses Limit muss sowohl
      in der ESP32-Firmware als auch im Backend erzwungen werden.
    - Im Backend: Beim Zuordnen von Karten zu einem Geraet pruefen, ob das
      Limit von 50 Karten erreicht ist. Falls ja, Fehlermeldung zurueckgeben.
    - Im Frontend: Anzeige der aktuellen Kartenanzahl pro Geraet mit Warnung
      ab 40 Karten ("80% Speicher belegt") und Blockade bei 50.
    - In der ESP32-Firmware: Karten werden in der NVS-Partition gespeichert
      ueber die Preferences-Bibliothek. Bei Ueberschreitung des Limits wird
      die aelteste unbenutzte Karte verdraengt.

1.6 Konfiguration auf dem ESP:
    - WiFi-SSID und Passwort (ueber seriellen Monitor oder WiFiManager
      Captive Portal konfigurierbar)
    - Server-URL (z.B. http://192.168.1.100:3001)
    - Device-ID (eindeutige Kennung, z.B. "esp32-door-01")
    - API-Key fuer Authentifizierung gegenueber dem Server
    - Tuer-Oeffnungsdauer in Sekunden
    - Anlernmodus-Timeout in Sekunden

2.2 Neue API-Endpunkte:

    AUTHENTIFIZIERUNG:
    - Alle /api/rfid/* Endpunkte muessen entweder per Session-Cookie (fuer die
      Weboberflaeche) ODER per API-Key Header (fuer den ESP32) authentifiziert
      werden
    - API-Key-Pruefung: Header "X-API-Key" gegen rfidDevices[].apiKey
    - Admin-Rechte fuer Verwaltungsendpunkte (Geraete/Karten CRUD)
    - ESP-Endpunkte (access, heartbeat) nur mit gueltigem API-Key

    GERAETE-VERWALTUNG (Admin-only):
    GET    /api/rfid/devices              - Alle Geraete auflisten
    POST   /api/rfid/devices              - Neues Geraet registrieren
                                            (generiert API-Key)
    GET    /api/rfid/devices/:id          - Einzelnes Geraet abrufen
    PUT    /api/rfid/devices/:id          - Geraet bearbeiten (Name,
                                            Oeffnungsdauer, Timeout etc.)
    DELETE /api/rfid/devices/:id          - Geraet loeschen
    POST   /api/rfid/devices/:id/mode     - Modus aendern (z.B. auf "learning")
                                            Body: {mode: "learning"} oder
                                            {mode: "normal"}
    POST   /api/rfid/devices/:id/regenerate-key - Neuen API-Key generieren

    KARTEN-VERWALTUNG (Admin-only):
    GET    /api/rfid/cards                - Alle Karten auflisten
                                            (mit Filtern: ?device_id=...,
                                             ?enabled=true)
    POST   /api/rfid/cards               - Neue Karte manuell registrieren
                                            Body: {uid, name, deviceIds}
    GET    /api/rfid/cards/:id            - Einzelne Karte abrufen
    PUT    /api/rfid/cards/:id            - Karte bearbeiten (Name, enabled,
                                            deviceIds, notes)
    DELETE /api/rfid/cards/:id            - Karte loeschen

    ESP32-ENDPUNKTE (API-Key-Auth):
    POST   /api/rfid/access              - Zugangspruefung
                                            Body: {uid, device_id}
                                            Response: {granted: true/false,
                                            card_name: "...", message: "..."}
    POST   /api/rfid/heartbeat           - Heartbeat/Status-Update
                                            Body: {device_id, firmware_version,
                                            ip_address, wifi_signal,
                                            free_memory, uptime}
    GET    /api/rfid/device/:id/mode     - Aktuellen Modus abfragen
                                            Response: {mode: "normal|learning",
                                            expires_at: "..."}
    GET    /api/rfid/device/:id/cards    - Kartenliste fuer dieses Geraet
                                            Response: [{uid: "AA:BB:CC:DD",
                                            enabled: true}, ...]
    POST   /api/rfid/device/:id/register-card  - Karte im Anlernmodus registrieren
                                            Body: {uid: "AA:BB:CC:DD"}

    ACCESS-LOGS (Admin-only):
    GET    /api/rfid/access-logs         - Access-Logs abrufen
                                            Query-Parameter:
                                            ?limit=100 (Standard)
                                            ?device_id=esp32-door-01
                                            ?uid=AA:BB:CC:DD
                                            ?action=ACCESS_GRANTED
                                            ?from=ISO-8601
                                            ?to=ISO-8601

2.3 Automatische Geraete-Offline-Erkennung:
    - Server prueft periodisch (alle 60 Sekunden), ob Geraete ihren Heartbeat
      verpasst haben (lastSeen aelter als 90 Sekunden)
    - Geraete-Status wird auf "offline" gesetzt, wenn kein Heartbeat kommt
    - Anlernmodus laeuft automatisch ab: Wenn modeExpiresAt ueberschritten,
      wird mode auf "normal" zurueckgesetzt
    - Optional: Log-Eintrag bei Status-Wechsel online->offline

2.4 Daten-Normalisierung:
    - Die bestehende normalizeData()-Funktion muss erweitert werden, um die
      neuen Arrays (rfidDevices, rfidCards, rfidAccessLogs) zu initialisieren
    - Access-Logs sollen auf maximal 2000 Eintraege begrenzt werden
    - Audit-Log-Integration: Wichtige RFID-Aktionen (Geraet erstellt, Karte
      hinzugefuegt, Karte geloescht, Modus geaendert) sollen auch in den
      bestehenden auditLogs erscheinen

5. SICHERHEITSANFORDERUNGEN
--------------------------------------------------------------------------------

5.1 API-Key-Sicherheit:
    - API-Keys werden serverseitig mit crypto.randomBytes(32) generiert
      und als Hex-String (64 Zeichen) ausgegeben
    - Keys werden nur einmal bei der Erstellung im Klartext angezeigt
    - In data.json werden die Keys als SHA-256-Hash mit Salt gespeichert.
      BEWUSST KEIN scrypt verwenden: Bei jedem Heartbeat (alle 30 Sekunden)
      und jedem Kartenscan muss der Server den Hash berechnen. scrypt ist
      dafuer auf einem Raspberry Pi zu rechenintensiv. SHA-256 mit Salt
      bietet fuer einen Prototyp im lokalen Netzwerk ausreichend Sicherheit
      bei deutlich besserer Performance.
      Implementierung:
        const salt = crypto.randomBytes(16).toString('hex');
        const hash = crypto.createHash('sha256')
          .update(salt + apiKey).digest('hex');
        // Gespeichert als: "sha256$<salt>$<hash>"
      Verifikation:
        const [algo, salt, storedHash] = device.apiKey.split('$');
        const checkHash = crypto.createHash('sha256')
          .update(salt + providedKey).digest('hex');
        return checkHash === storedHash;
    - ESP32 sendet den Key im Header: "X-API-Key: <key>"

5.2 Eingabevalidierung:
    - UID-Format validieren (hexadezimal, korrekte Laenge)
    - Device-ID: Nur alphanumerisch + Bindestriche, 3-50 Zeichen
    - Name: Max 100 Zeichen, kein HTML

5.2b Rate-Limiting (ESP-Endpunkte):
    - Max 60 Requests pro Minute pro Device-ID
    - Mechanismus: Einfacher In-Memory-Counter pro Device-ID mit Zeitfenster-
      Reset. Implementierung als Object/Map im Server:
        const rateLimits = {};  // { "esp32-door-01": { count: 12, resetAt: timestamp } }
      Bei jedem ESP-Request:
        1. Device-ID aus dem Request lesen
        2. Wenn kein Eintrag oder resetAt < Date.now(): neuen Counter starten
           mit count=1 und resetAt = Date.now() + 60000 (1 Minute)
        3. Wenn count >= 60: HTTP 429 "Too Many Requests" zurueckgeben
        4. Sonst: count um 1 erhoehen und Request verarbeiten
    - Dieses einfache Verfahren reicht fuer den Prototyp voellig aus und
      benoetigt keine externe Bibliothek.

5.3 Zugriffskontrolle:
    - Alle Verwaltungsendpunkte nur fuer ADMIN-Rolle
    - ESP-Endpunkte nur mit gueltigem API-Key
    - Access-Logs nur fuer ADMIN einsehbar
    - Keine sensiblen Daten in Client-Responses (keine API-Keys in GET-Antworten)

5.4 Hinweis:
    - Da es sich um ein Schulprojekt/Prototyp handelt, sind folgende Punkte
      bewusst vereinfacht:
    - Keine TLS/HTTPS-Erzwingung (lokales Netzwerk)
    - Keine verschluesselten API-Keys auf dem ESP (im NVS gespeichert)
    - Kein TOTP/2FA fuer den Tuerzugang

6.2 Verwendete Libraries (explizit festgelegt):
    Die folgenden Libraries sollen verwendet werden. KEINE Alternativen
    waehlen, um Konsistenz und Eindeutigkeit sicherzustellen:

    RFID-Reader:
    - Library: MFRC522 von Miguel Balboa (miguelbalboa/MFRC522)
      GitHub: https://github.com/miguelbalboa/rfid
      Installation: Arduino Library Manager -> "MFRC522" suchen
      #include <MFRC522.h>
      Begruendung: De-facto-Standard fuer MFRC522 mit Arduino/ESP32,
      umfangreiche Dokumentation, aktive Community.

    HTTP-Client:
    - Library: HTTPClient (im ESP32-Arduino-Core enthalten)
      #include <HTTPClient.h>
      Begruendung: Keine zusaetzliche Installation noetig, bereits im
      ESP32-Core integriert, unterstuetzt GET/POST/PUT/DELETE.

    NVS-Kartenspeicherung:
    - Library: Preferences (im ESP32-Arduino-Core enthalten)
      #include <Preferences.h>
      Begruendung: Einfache Key-Value-API fuer den NVS-Speicher,
      keine zusaetzliche Installation noetig, persistiert ueber Resets.

    WiFi:
    - Library: WiFi (im ESP32-Arduino-Core enthalten)
      #include <WiFi.h>
      Optional fuer Captive-Portal-Setup: WiFiManager von tzapu
      #include <WiFiManager.h>

    SPI (fuer MFRC522):
    - Library: SPI (Built-in)
      #include <SPI.h>

    JSON-Verarbeitung:
    - Library: ArduinoJson von Benoit Blanchon (bblanchon/ArduinoJson)
      #include <ArduinoJson.h>
      Installation: Arduino Library Manager -> "ArduinoJson" suchen
      Begruendung: Effizientes JSON-Parsing und -Serialisierung,
      speziell optimiert fuer Mikrocontroller mit begrenztem RAM.

    ZUSAMMENFASSUNG der #include-Direktiven im Haupt-Sketch:
    #include <SPI.h>             // Built-in: SPI-Bus fuer MFRC522
    #include <MFRC522.h>         // Extern: RFID-Reader
    #include <WiFi.h>            // Built-in: WiFi-Verbindung
    #include <HTTPClient.h>      // Built-in: HTTP-Requests
    #include <Preferences.h>     // Built-in: NVS-Speicher
    #include <ArduinoJson.h>     // Extern: JSON-Verarbeitung

7. WICHTIGE IMPLEMENTIERUNGSHINWEISE
--------------------------------------------------------------------------------

7.1 Backend (server-simple.js):
    - Der Server nutzt KEIN Express-Framework, sondern raw Node.js http.createServer
    - Routing erfolgt manuell ueber URL-Matching in der handleAPI()-Funktion
    - Neue Endpunkte muessen in derselben Funktion ergaenzt werden
    - JSON-Body-Parsing erfolgt ueber die bestehende parseBody()-Funktion
    - Authentifizierung: Bestehende requireAuth() und requireAdmin() nutzen,
      PLUS neue requireApiKey()-Hilfsfunktion fuer ESP-Endpunkte
    - Daten werden in der globalen "data"-Variable gehalten und periodisch
      in data.json gespeichert (dataChanged-Flag setzen!)

7.2 Frontend (script.js):
    - Das Frontend ist eine Single-Page-Application in Vanilla JavaScript
    - Es gibt KEIN Framework (kein React, kein Vue)
    - Neue UI-Elemente werden als HTML-Strings oder per DOM-Manipulation erstellt
    - Das Admin-Panel wird dynamisch generiert
    - Bestehende Patterns fuer Modals, Tabellen und Formulare wiederverwenden
    - API-Aufrufe erfolgen ueber fetch() mit credentials: 'include'
    - Neue Funktionen als Methoden der bestehenden Klassen oder als
      eigenstaendige Funktionen im globalen Scope hinzufuegen

7.3 HTML (index.html):
    - Neue Abschnitte fuer das Admin-Panel als versteckte Divs hinzufuegen
    - Navigation im Admin-Bereich um RFID-Tabs erweitern
    - Modal-Templates fuer Geraete- und Kartendialoge hinzufuegen

7.4 Reihenfolge der Implementierung (empfohlen):
    1. Datenstrukturen und Normalisierung in server-simple.js
    2. Backend-API-Endpunkte (CRUD fuer Devices und Cards)
    3. ESP32-spezifische Endpunkte (Access, Heartbeat, Mode, Sync)
    4. Access-Log-Endpunkte
    5. Frontend: Admin-Panel Navigation und Tab-Struktur
    6. Frontend: Geraete-Verwaltung UI
    7. Frontend: Karten-Verwaltung UI
    8. Frontend: Access-Log-Viewer
    9. Frontend: Anlernmodus mit Countdown und Echtzeit-Feedback
    10. ESP32-Firmware
    11. Integration & Tests
    12. Audit-Log-Integration

7.5 Testbarkeit & Simulationsmodus:
    - Die Backend-API sollte mit curl oder Postman testbar sein, auch ohne
      echten ESP32
    - Device-Status kann manuell ueber die API gesetzt werden

    SIMULATIONSMODUS IM FRONTEND:
    - In der Geraete-Verwaltung soll pro Geraet ein aufklappbarer Bereich
      "Scan simulieren" vorhanden sein (nur sichtbar im Admin-Panel).
    - Dieser Bereich enthaelt:
      a) Ein Textfeld zur Eingabe einer UID (Platzhalter: "AA:BB:CC:DD",
         Validierung auf hexadezimales Format mit Doppelpunkt-Trennung)
      b) Einen Button "Scan ausfuehren", der den eingegebenen Wert per
         POST /api/rfid/access {uid: "<eingegebene-uid>", device_id: "<geraet-id>"}
         an die API sendet
      c) Einen Button "Zufaellige UID", der eine zufaellige 4-Byte-UID
         generiert und ins Textfeld eintraegt (zum schnellen Testen
         unbekannter Karten)
      d) Ein Ergebnis-Feld, das die Server-Antwort anzeigt:
         - Gruen: "Zugang gewaehrt fuer <Kartenname>"
         - Rot: "Zugang verweigert - Karte unbekannt/deaktiviert"
         - Blau: "Karte registriert" (wenn Geraet im Anlernmodus)
    - Dieser Simulationsmodus erlaubt vollstaendiges Testen des gesamten
      Flows (Zugang, Ablehnung, Anlernmodus) ohne physische RFID-Hardware.
